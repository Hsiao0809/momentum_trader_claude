# 50 — 制度檔維護協議（弱模型如何安全地更新這套檔案）

## 0. 通則

- 改任何 `.claude/` 下的檔案或 `CLAUDE.md` **之前**：`cp 原檔 .claude/backups/檔名.YYYY-MM-DD.bak`（目錄不存在就建）。備份檔要進 git、和修改一起 commit，這樣任何 session 都能還原。
- 制度檔的修改和 code 改動一樣走分支 + PR，commit message 前綴 `docs(claude):`。
- 一次只改一個檔的一個主題；不要「順手大整理」。

## 1. 權限分級

**可以自行改（不必問使用者）**：
- `LESSONS.md` 追加教訓（見 §2；追加 = 只加不刪）。
- `10-PROJECT-MAP.md` 的**事實更新**：函式增減、行號漂移、指令變動、新增檔案。判準：改的是「repo 現在長什麼樣」的描述，而不是「該怎麼做事」的規則。
- 修正錯字、失效路徑、失效工具名（修正後要在 commit message 說明原值為何失效）。

**先問使用者才能改**（用 AskUserQuestion 給選項）：
- CLAUDE.md 的硬規則、`20-DISPATCH.md` 的升降級路徑、`30-JUDGMENT.md` 的判準本身——這些是制度核心，改弱一條就是全面漏水。**特別是：任何「放寬」（把必跑改成建議跑、把禁止改成盡量避免）一律要使用者同意。**
- 刪除任何既有規則或整段內容。
- 新增會約束未來所有 session 的新規則（新增判準比放寬安全，但仍要問——規則膨脹也是退化，見 60-LETTER）。
- 修改 `.claude/agents/` 下的 agent 定義。

**任何情況都不可做**：
- 因為「這條規則讓我現在的任務變麻煩」而繞過或臨時修改規則。規則擋路時的正確動作：照規則做，或問使用者。

## 2. LESSONS.md：踩坑教訓寫回處

- 位置：`.claude/docs/LESSONS.md`，**append-only**（日常只加不刪；唯一例外是下述「精簡」，屬「先問使用者」級），新條目加在最上方。
- **什麼算「值得寫」**：花了你超過兩輪嘗試的坑、或違反直覺的環境事實、或驗收抓到的真 bug 模式。普通打錯字不寫。
- 格式（固定四行，禁止長篇）：

```
## YYYY-MM-DD 一句話標題
坑：當時發生什麼（含錯誤訊息關鍵字，方便未來 grep）。
因：真正原因。
避：下次怎麼做（可執行的一句話）。
```

- **何時精簡**：條目超過 30 條或檔案超過 300 行時，發起一次精簡 PR：重複主題合併、已寫進正式制度檔的條目刪除（並在對應制度檔加上該規則）、超過一年未再踩的坑移到檔尾「封存」段。精簡屬於「先問使用者」級操作。

## 3. 每季健檢（或使用者要求時）

跑一次以下清單，結果回報使用者：
1. `10-PROJECT-MAP.md` 抽查 5 個函式名 grep 是否還存在、指令表逐條可跑。
2. `20-DISPATCH.md` §0 的模型 enum 與 agent 清單，對照當下 system prompt 是否仍準確。
3. `LESSONS.md` 是否到達精簡門檻。
4. CLAUDE.md 是否超過 60 行（超過 = 開始膨脹，把細節下放到 docs）。

## 4. 衝突處理

- 任何矛盾（制度檔之間、或與使用者指示之間）：照 CLAUDE.md 開頭那張優先序表處理（全制度唯一一張，本檔不另立），先照高優先者做，然後回報使用者並提議修掉矛盾（修矛盾屬於「先問」級修改）。
