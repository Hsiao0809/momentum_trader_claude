# 30 — 判斷力 rubric（拿不定主意時，照這裡判，不要憑感覺）

> 每條判準都附正例（✅ 該這樣）與反例（❌ 不該這樣）。例子多半用本 repo 的實際場景。

## 1. 何時該升級模型（或求援）

**判準**：滿足任一條就按 `20-DISPATCH.md` §4 升級：
(a) 同一子任務你已失敗兩輪且第二輪沒有新資訊；(b) 問題需要跨檔推理時序/併發（cron lock、Queue 重試、KV 競態）；(c) 你發現自己在「試試看這樣行不行」而說不出為什麼會行。

- ✅ 正例：改 `runPaperTick` 的 lock 邏輯，兩次嘗試後測起來仍偶爾重複開倉。停止再試，派 `second-opinion`（opus）並附上兩次的 diff、觀察到的現象、已排除的假設。
- ❌ 反例：Edit 因為 old_string 比對失敗了兩次就升級模型。這不是能力問題，是方法問題——換更短的錨點字串或用行號編輯即可（見 `00-DIAGNOSIS.md` 弱點 2 修法 4）。

## 2. 何時算「真的完成」

**判準**：以下全部成立才能對使用者說完成：
(a) 使用者原話要求的每一項都有對應改動或明說為何不做；(b) `npm run verify` 通過且結果貼在回覆裡；(c) 策略類改動已做雙實作檢查並明說結論；(d) 動到 tick/掃描/KV 路徑的，已逐條核對 PROJECT-MAP 硬性約束並寫出預算影響；(e) 非 trivial 改動已由 `acceptor` 驗收 PASS。

- ✅ 正例：「把 TP1 從 +20% 改成 +18%」→ 兩檔的 `takeTP1`/相關常數都改、verify 通過、acceptor 確認兩檔數值一致，回覆附兩處 檔案:行號。
- ❌ 反例：只改了 worker 的常數，語法沒驗，回覆「已完成，dashboard 應該也會跟著顯示正確」。「應該會」= 沒驗 = 沒完成。

**「動到但沒說」也算沒完成**：改動範圍超出使用者要求的部分（如順手重排了函式），必須在回覆中列出，否則視同未經同意的暗改。

## 3. 何時該停下來問使用者

**判準**：滿足任一條就停，用 AskUserQuestion 給 2–4 個具體選項（每個選項附後果），不要開放式反問：
(a) 改動會變更**交易行為**（開倉條件、停損停利、部位大小、掃描頻率）而使用者沒有明確指定數值/規則；(b) 需求有兩種以上合理解讀且做錯的返工成本高；(c) 動作不可逆或影響線上（部署、刪 KV 資料、reset 狀態）；(d) 你即將修改使用者沒提到的既有行為。

- ✅ 正例：「訊號太多了，過濾一下」→ 停：這是策略決策。問「要提高 anomalyScore 門檻 / 減少候選數上限 / 加最低成交量過濾？各自的效果是＿＿」。
- ❌ 反例：「把平倉表格加上排序」→ 這是純 UI、可逆、無歧義，直接做完展示即可，不要問「請問要升冪還是降冪」這種可以先做預設再讓使用者改的問題。

**不問的預設**：可逆、純工程、不動交易行為 → 選最合理的預設做完，並在回覆中標明你做的假設。

## 4. 什麼訊號代表方向錯了（該換路，不是再試一次）

**判準**：出現任一訊號，停止當前路線，寫下「目前假設是什麼、哪個證據推翻了它」，再選新路：
(a) 修好 A 弄壞 B、修好 B 又弄壞 A（打地鼠循環）；(b) 為了讓方案成立，你需要的「例外」越來越多；(c) 第二輪重試的 diff 和第一輪幾乎一樣；(d) 你開始懷疑是工具/環境壞了而不是自己的假設錯了。

- ✅ 正例：想在掃描路徑直接發通知，一直撞 subrequest 限制，開始想「分兩批發、加 sleep」——停。例外越補越多就是方向錯了：README 明說這正是通知走 Queue 的原因，正路是用 `NOTIFICATION_QUEUE`。
- ❌ 反例：`npm run verify` 因為改壞一個括號而失敗，這不是「方向錯了」，是普通 bug，修掉即可。換路判準針對的是**連續、結構性**的失敗，不是單次錯誤。

## 5. 品質底線怎麼驗（最低標準，不可再降）

任何 code 改動，底線由低到高：

1. **語法**：`npm run verify` 通過（永遠要求）。
2. **行為**：改動的函式至少被實跑過一次——worker 邏輯可用一次性 node script `import` 純函式餵假資料；回測/統計類可在瀏覽器 console 或 node 跑（純函式居多）。做不到實跑時，必須在回覆明說「僅靜態驗證」。
3. **獨立驗收**：非 trivial 改動派 `acceptor`（見 `20-DISPATCH.md` §5）。
4. **對外**：交 PR 前跑 `/code-review`。

- ✅ 正例：改了 `atrPct`，寫 10 行臨時 script 餵一段固定 K 線，確認新舊輸出差異符合預期，然後刪掉臨時 script。
- ❌ 反例：「我讀過一遍 code，邏輯正確」。讀 code 不是驗證，是猜測。

## 6. 誠實條款（rubric 的極限）

- 以上判準覆蓋工程執行，**不覆蓋交易策略的好壞**。凡「這樣改會不會更賺」→ 跑 `runBacktest` 給數字或明說無法評估，永遠不要給出「這樣比較好」的策略意見當作事實。
- 判準之間衝突時的優先序：不弄壞線上 runner > 使用者明確指示 > 本檔判準 > 省 token。
- 遇到本檔沒覆蓋的判斷且風險高：升級模型求第二意見；還是不確定就問使用者，並誠實說「我不確定」。
