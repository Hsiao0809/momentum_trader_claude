# 40 — 派工 prompt 模板（複製、填空、發出；不要即興發揮）

> 用法：選對應模板 → 填 `【】` → 照 `20-DISPATCH.md` §0 選 agent 類型與 model → 發出。
> 所有模板已內建三件套（目標與動機／驗收條件／回報格式）與回報合約。**刪掉任何一段都算違規派工。**

## 通用尾註（每個模板結尾都保留）

```
本 repo 須知：先讀 .claude/docs/10-PROJECT-MAP.md 的「雙實作對照表」與「硬性外部約束」。
momentum_trader_claude.html 與 worker/src/index.js 禁止無 offset 整檔 Read，先 Grep 再讀區段。
回報守則：不要把大段檔案內容貼回來；長產物寫到【落檔路徑】，回報只給路徑＋三行摘要。
```

## T1 搜尋／定位（agent: Explore；無需 model 覆寫）

```
目標：找出【要找什麼，例：所有會寫入 KV 的呼叫點】。
動機：【為什麼找，例：評估新增一次 saveState 對每日寫入預算的影響】。
搜尋廣度：【medium / very thorough】。
範圍：【檔案或目錄，例：worker/src/index.js】。
驗收條件：
- 每個結果附 檔案:行號 與該行原文。
- 明說搜了哪些 pattern，以及為什麼可以相信沒有遺漏（例：也搜了別名/縮寫）。
回報格式：結論一句 → 結果清單（每行「檔案:行號 — 一句說明」）→ 搜索覆蓋說明。
＋通用尾註
```

## T2 實作（agent: general-purpose；model: sonnet，機械套用時 haiku）

```
目標：【要做的改動，含具體數值/行為】。
動機：【使用者原話或需求背景，一句】。
相關位置：【檔案:大約位置或函式名】。
邊界：只改【範圍】；不動【明確排除項，例：策略參數、部署設定】。
驗收條件（逐條可機械判定）：
- 【例：兩檔 buildRisk 的 TP1 常數共四處皆已改（多單 1.20→1.18、空單鏡像 0.80→0.82；HTML 寫法 `tp1:entry*1.20`、worker 寫法 `tp1: entry * 1.20`），grep 可證】
- npm run verify 通過（貼輸出末三行）。
- 若動到 tick/掃描/KV 路徑：寫出每次 tick 的 KV write 次數與外部 fetch 次數變化（+N/-N/不變）。
回報格式：結論一句 → 逐條驗收自查結果 → 改動點清單（檔案:行號）→ 你做的假設（沒有寫「無」）。
＋通用尾註
```

## T3 重構（agent: general-purpose；model: sonnet；動手前建議先派 Plan）

```
目標：【重構什麼，例：把 HTML 內重複的 K 線工具函式集中到單一區塊】。
動機：【為什麼值得，例：三處重複導致上次改漏】。
不變式（重構後必須完全不變的行為）：
- 【例：runBacktest 對同輸入輸出相同】
- 【例：worker 檔案完全不動】
驗收條件：
- npm run verify 通過。
- 不變式逐條驗證：【怎麼驗，例：重構前先跑一次 X 記下輸出，重構後再跑比對】。
- diff 中不得出現任何行為改變（只有搬移/改名/去重）；有就中止並回報。
回報格式：結論一句 → 不變式驗證證據 → 改動點清單（檔案:行號）→ 中途發現但沒動的問題清單。
＋通用尾註
```

## T4 研究／查證（agent: general-purpose；model: sonnet；需要 WebSearch）

```
問題：【要回答什麼，例：Cloudflare Workers free plan 目前每請求 subrequest 上限】。
動機：【這個答案會決定什麼，例：是否能在 tick 內多打 5 次 OKX】。
要求：
- 優先官方文件/一手來源；每個關鍵結論附來源 URL 與查閱日期。
- 區分「文件明說」與「你的推論」，分開標示。
- 查不到就寫查不到，禁止用訓練記憶充當查證結果。
驗收條件：每個結論有來源；互相矛盾的來源要並列並說明取捨。
回報格式：直接答案（≤3 行）→ 依據（來源＋引句）→ 不確定事項。
＋通用尾註
```

## T5 審查／驗收（agent: acceptor；高風險方案改派 second-opinion）

```
待驗改動：【一句描述】＋【涉及檔案清單】。
改動意圖：【原需求一句】。
驗收條件（逐條，含驗法）：
- 【例：momentum_trader_claude.html 與 worker/src/index.js 的 effectiveStopFor 邏輯一致 — 兩檔各 Read 該函式比對】
- 【例：npm run verify 通過 — 實跑】
- 【例：tick 路徑 KV write 次數不變 — Read runPaperTick 數 saveState 呼叫】
回報格式：照你 agent 定義檔的格式（總判定 → 逐條證據 → 最小修復建議）。
```

## 常見錯誤（派工前自查）

1. **忘了 subagent 是冷啟動**：模板裡沒提的背景它一概不知。與任務相關的使用者原話、先前的失敗嘗試，要貼進 prompt。
2. **驗收條件寫成願望**：「確保正確」「品質良好」不可機械判定，等於沒寫。改成「grep 得到／跑得過／兩檔一致」。
3. **升級重派時不附失敗軌跡**：升級 prompt 必附前幾輪的 diff、錯誤訊息原文、已排除假設（`20-DISPATCH.md` §4）。
4. **一個 prompt 塞多個目標**：一個 subagent 一件事。兩件事就派兩次（可平行）。
